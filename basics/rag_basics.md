# RAG  Basics:

## Phase 1 & 2: Naive Ingestion, Structural Chunking, and Local Persistence

The foundational RAG pipeline relies on a stateless flow:

```text
Ingest → Chunk → Embed → Store → Retrieve → Generate
```

To eliminate repetitive API overhead, raw data metrics are converted into unified vector layouts and cached permanently inside persistent storage directories.

### Structural Logic Check

#### Directory Ingestion

Read the raw text strings out of your local source folders via `SimpleDirectoryReader`.

#### Identifier Assignment

Map the exact file paths as absolute database keys using `filename_as_id=True`.

#### Persist Strategy

If the local cache folder is absent:

1. Parse files.
2. Call the embedding API to calculate vectors.
3. Execute `.persist()`.

#### Sync / Refresh Optimization

If the storage folder is already on disk, skip full indexing.

Hydrate the data into memory and run a `.refresh_ref_docs()` loop. This checks cryptographic string hashes, or digital fingerprints, of local files against the database metadata.

If a hash matches, processing is skipped with `$0` API cost.

If a hash mismatches, the old chunks are dropped and updated dynamically.

### Core Ingestion Parameters

| Parameter                | Description                                                                                                                                              |
| ------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `Settings.chunk_size`    | The hard character/token limit ceiling defining the maximum scale of an isolated text fragment.                                                          |
| `Settings.chunk_overlap` | The token boundary padding shared between adjacent chunks. This prevents critical sentences from being sliced down the middle across two separate nodes. |

### Configuration Snippet

```python
Settings.chunk_size = 512      # Optimized for strict semantic precision
Settings.chunk_overlap = 50    # Preserves structural continuity across split lines

# Core storage persistence check and execution
storage_context = StorageContext.from_defaults(persist_dir="./storage")
index = load_index_from_storage(storage_context)

# Sync folder updates to local state
refreshed_docs = index.refresh_ref_docs(documents)
if any(refreshed_docs):
    index.storage_context.persist(persist_dir="./storage")
```

## Phase 3 & 4: Multi-Query Expansion, Hybrid Search, and Two-Stage Reranking

Pure semantic vector space search struggles to find exact keyword tokens, product serial numbers, error codes, and unique jargon. To address this limitation, high-performance engines run a two-stage parallel search matrix.

### The Parallel Retrievers

#### Dense Retrieval: Semantic Space

Processes text via multi-dimensional vectors to lock onto structural concepts, meanings, and synonyms.

#### Sparse Retrieval: BM25 Algorithm

Processes character-level patterns, checking exact keyword occurrences weighted by term frequency and inverse document frequency.

### The Reciprocal Rank Fusion, or RRF, Algorithm

Because dense vector similarities, measured by cosine distance coefficients, and sparse character matches, measured by BM25 scoring tables, generate completely different numerical value ranges, raw scores cannot be added together.

Instead, the scores are stripped away, and candidates are evaluated purely by their relative rank position across both parallel response lists using this reciprocal calculation formula:

```math
RRF(d) = \sum_{m \in M} \frac{1}{k + \text{rank}_m(d)}
```

Where:

| Symbol      | Meaning                                                                |
| ----------- | ---------------------------------------------------------------------- |
| `M`         | The list of distinct retrieval search engines deployed.                |
| `rank_m(d)` | Document `d`'s ordinal position inside retriever list `m`.             |
| `k`         | An architectural smoothing constant, traditionally defaulting to `60`. |

### Two-Stage Retrieval Mechanics

#### Stage 1: The Net

Pull a broad, dense candidate pool, such as the top 10 matches, across multiple query variations generated by an internal LLM paraphraser.

#### Stage 2: The Filter

Feed those candidates through a Cross-Encoder Reranker.

Unlike standard vectors that process queries and documents in isolation, a reranker passes the query and document together through a deep neural model to compute absolute contextual relevance.

This filters out the noise and trims the selection down to the top 3 best chunks.

### Core Retrieval Parameters

| Parameter           | Description                                                                                                                                                          |
| ------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `similarity_top_k`  | Dictates the initial candidate harvest volume size returned per individual database lookup action.                                                                   |
| `num_queries`       | Regulates query variations. Setting this to `N > 1` instructs the LLM to rewrite the user's string into multiple distinct versions to catch varying phrasing angles. |
| `choice_batch_size` | Limits the number of nodes evaluated concurrently by the reranking layer to protect memory buffers.                                                                  |
| `top_n`             | The ultimate structural bottleneck constraint. Trims the context bundle down to the absolute best snippets right before wrapping them into the final prompt.         |

### Configuration Snippet

```python
# Stage 1 Retrieval Configuration
vector_retriever = index.as_retriever(similarity_top_k=10)
bm25_retriever = BM25Retriever.from_defaults(nodes=nodes, similarity_top_k=10)

# Multi-Query Paraphraser & RRF Fusion Setup
hybrid_retriever = QueryFusionRetriever(
    retrievers=[vector_retriever, bm25_retriever],
    similarity_top_k=10,            # Fuses the parallel arrays down to 10 unique elements
    num_queries=4,                  # Automated 4-way query expansion
    mode="reciprocal_rerank"        # Apply the RRF positional formula
)

# Stage 2 Filtering Control
reranker = LLMRerank(choice_batch_size=5, top_n=3)
```

## Phase 5: Autonomous Agentic Engineering with the Workflows API

Modern AI systems, including LlamaIndex v0.13+, deprecate linear, rigid chat pipelines. Instead, they wrap retrieval components inside standalone utilities called Tools, giving execution control to an autonomous Workflow Agent.

### The ReAct Execution Framework

The agent acts as an independent loop controller processing text inputs using a transparent reasoning loop:

```text
User Input → Thought → Action: Select Tool → Action Input → Observation: Result → Final Answer
```

The LLM evaluates the user's intent:

* If a simple greeting is given, it skips tools entirely.
* If a calculation is requested, it calls a function tool.
* If deep document synthesis is requested, it routes execution to your hybrid RAG tool.

### Agentic Structural Concepts

| Concept             | Description                                                                                                                         |
| ------------------- | ----------------------------------------------------------------------------------------------------------------------------------- |
| `ReActAgent`        | The parent workflow container coordinating the asynchronous reasoning loops.                                                        |
| `Context`, or `ctx` | An isolated, thread-safe state manager tracking user historical data across the application runtime without risking memory leakage. |
| `user_msg`          | The mandatory explicit keyword signature property required by modern async runtimes to ensure strict argument mapping.              |

### Configuration Snippet

```python
# Package your advanced query components into an isolated Tool object
document_tool = QueryEngineTool.from_defaults(
    query_engine=query_engine,
    name="document_search_tool",
    description="Locates exact historical facts, data summaries, and specific text matches."
)

# Instantiating the Workflow elements
agent = ReActAgent(tools=[document_tool, calculator_tool], llm=Settings.llm)
ctx = Context(agent)

# Execute via modern asynchronous keyword signature format
response = await agent.run(user_msg=user_input, ctx=ctx)
```

## Phase 6: Automated Pipeline Evaluation Metrics

To optimize your application scientifically rather than guessing whether adjustments improve performance, deployments are continuously audited using the RAG Triad.

### The Validation Metrics

| Metric                        | Description                                                                                                                                                                                             |
| ----------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Faithfulness, or Groundedness | Cross-examines the final generated answer strictly against the retrieved context nodes. If the model introduces concepts or assertions absent from the context, the test flags a hallucination failure. |
| Answer Relevancy              | Quantifies whether the output directly matches the core user intent or drifts off into noise.                                                                                                           |
| Context Recall                | Validates the retriever indices, verifying whether the system effectively captured all essential facts buried within the master files.                                                                  |

### Configuration Snippet

```python
# Initialize objective evaluation judges
# Enforce 0.0 temperature for consistency
faithfulness_evaluator = FaithfulnessEvaluator(
    llm=OpenAI(model="gpt-4o-mini", temperature=0.0)
)

# Execute the validation algorithm
faith_result = faithfulness_evaluator.evaluate_response(response=response_object)

print(f"PASS Status: {faith_result.passing} | Score: {faith_result.score}")
```
